# Roman Numeral Converter – Report on Generation Process and Quality Assessment

> Note: This essay was made by Juan Ramirez (alu.163243) and Pablo Villarte (alu.156336).

## 1. Introduction  

The objectives of this essay are to develop a website capable of converting Roman numerals to integers and vice versa, and to perform a quality assessment based on four different quality definitions. To create the website, a set of functional and non‑functional requirements was first established. Artificial intelligence was then used to generate the code through a specifically designed prompt aligned with those requirements. Subsequently, a manual quality analysis was conducted to evaluate the generated code according to four quality perspectives: conformance to requirements, user‑centered quality, internal and external quality, and the ISO 9126 quality model.

---

## 2. Requirements Defined  

### Functional Requirements  

- Convert integers (1–3999) to Roman numerals.  
- Convert Roman numerals to integers.  
- Validate:  
  - Empty input  
  - Invalid characters  
  - Invalid Roman numeral patterns  
  - Out‑of‑range integers  
- Display clear error messages.  
- Reset previous results before each conversion.

### Non‑Functional Requirements  

- Responsive layout for mobile, tablet, and desktop.  
- Accessible UI with labels, contrast, and keyboard support.  
- Clean code following SOLID principles.  
- Separation between conversion logic and UI logic.  
- Maintainable structure suitable for testing and CI/CD.

---

## 3. Tools and Prompt Used  

### Tools used  

- ChatGPT for code generation.  
- Visual Studio Code for reviewing the code files and markdown editing.  

### Prompt Used  

> Generate a simple, responsive Roman numeral converter webpage that follows clean code and SOLID principles, with HTML, CSS, and JavaScript in separate files; the JavaScript must clearly separate pure, test‑ready conversion logic (integer to Roman, Roman→ to integer, and strict Roman numeral validation) from UI code so the logic is not coupled to the UI; the UI should be minimal, centered, mobile/tablet/desktop friendly, and easy to use, with good accesibility and ease of use. The code should be self explanatory through naming and structure rather than comments; the page must support conversion in both directions, validate all inputs (invalid characters, invalid Roman patterns, empty input, out‑of‑range integers), reset previous results before each conversion, and be structured in a way that makes future refactoring, unit testing, automation, and CI/CD easy to apply.  

---

## 4. Quality Assessment  

### View 1: Conformance to Requirements  

The system generated by the AI follows the requirements defined for the project in a consistent way. Both conversion directions work as expected, and the validation rules are applied strictly, preventing incorrect Roman numeral formats and out‑of‑range integers. The interface remains simple and responsive, and the structure of the project keeps the different layers separated. These aspects can be validated from both a functional and non-functional perspective.

From a functional perspective:

- Integer → Roman and Roman → integer conversions work correctly within the defined range (1–3999).
- Strict validation prevents invalid characters, incorrect subtractive patterns, illegal repetitions, and non-canonical Roman numerals.
- Clear error messages are displayed for invalid or empty input.

From a non-functional perspective:

- The interface is responsive across devices.
- Accessibility elements (labels, ARIA attributes, focus behavior) are properly implemented.
- The architecture separates logic from UI, which aligns with good development standards and maintainability expectations.

Therefore, the implementation demonstrates strong conformance to both explicit and implicit requirements. Under this view, the quality of the system can be considered high.

### View 2: User‑Centered Quality  

Under the human-centered definition of quality, software quality is determined by how users perceive and experience the system. From the user’s perspective, the website is intuitive and easy to understand. The interface is clean and avoids unnecessary visual clutter, allowing users to focus on the core functionality. Immediate feedback helps users quickly determine whether their input is valid, improving clarity and confidence in the system. Additionally, the design adapts effectively to different screen sizes.

Some relevant aspects include:

- A minimal and readable interface.  
- Clear error messages and status updates.  
- Good behaviour on mobile, tablet, and desktop devices.  
- Accessibility support through ARIA labels, focus indicators, and reduced‑motion options.  

However, some users may expect additional customization options, such as a light/dark theme toggle, since many modern websites provide this feature. In the current implementation, the application offers only a dark blue theme, which may not suit every user’s preference. Furthermore, increased contrast between certain background and foreground elements could improve readability for some users.

These limitations do not significantly affect usability but represent potential areas for enhancement. Importantly, such improvements could be implemented relatively easily due to the system’s modular internal structure, as discussed in the next quality perspective.

Overall, these characteristics contribute to a user experience that is straightforward, accessible, and comfortable.

### View 3: Internal vs. External Quality  

From an internal perspective, the system is structured in a way that enhances readability, maintainability, and extensibility. The conversion logic is implemented as pure functions that return structured results, while the user interface logic is handled separately. This clear separation of concerns reduces coupling and facilitates future modifications, testing, and refactoring.

From an external perspective, these internal design decisions translate into consistent and reliable behaviour. The system performs accurate conversions, enforces strict validation rules, and provides predictable responses to both valid and invalid inputs. Users receive immediate and clear feedback, reinforcing trust in the system.

This relationship demonstrates how strong internal quality directly supports and sustains external quality over time.

**Technical Debt Consideration:**  
Although the code is generally clean and well‑structured, some technical debt is still present, mainly because the system is limited to the 1–3999 range and relies on custom validation logic that may need updates if requirements change. However, the modular design helps keep this debt under control, making future improvements easier to implement.

### View 4: ISO 9126 Quality Attributes  

The ISO 9126 model provides a structured framework to evaluate software quality across multiple dimensions. Using the ISO 9126 model, the system performs well across the main quality attributes:

| Attribute | Evaluation |
| ---------- | ------------ |
| **Functionality** | Accurate conversions with strict validation rules. |
| **Reliability** | Predictable behaviour and clear error handling. |
| **Usability** | Simple layout, accessible controls, and immediate feedback. |
| **Efficiency** | Fast execution with minimal resource usage. |
| **Maintainability** | High, thanks to modular design and separation of concerns. |
| **Portability** | Works in any modern browser without installation. |

The system shows balanced performance across these attributes, with particular strengths in maintainability, usability, and functionality.
